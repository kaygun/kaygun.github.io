The Kitchen Sink and Other Oddities

Atabey Kaygun

From Magmas and Catamorphisms to Coalgebras of Computation

In the previous post, I described how the syntax of applicative calculi can be organized by magmas and catamorphisms. The guiding idea was simple: application is binary, so application syntax naturally gives binary trees. A program is built by constructors, and a syntax-directed transformation is a fold over that inductive structure.

That is the algebraic half of the story.

In this post I want to describe the coalgebraic half. Algebra builds syntax. Coalgebra unfolds behaviour. A catamorphism consumes an abstract syntax tree. A coalgebraic semantics observes a running program as a state and asks: what can this state do next?

The distinction is important. A lambda term, an SKI term, or an iota term can be viewed syntactically as a tree. But a program is not merely a tree. Once reduced to a computational state, the program can receive arguments, expose a head form, and continue to another state. That transition structure is not best described by a fold. It is best described by a coalgebra.

From syntax to reduced states

Let us begin with the algebraic side. A formal applicative language has a collection of program texts and a binary operation of application. In the simplest combinatory cases, syntax is literally a free magma.

For example, an SKI term is generated by

M ::= S | K | I | M M

so the syntax is the free magma on the set {S, K, I}. In Scala one might model this as follows.

enum SKI:
  case S
  case K
  case I
  case App(left: SKI, right: SKI)

The constructor App is the magmatic product. A fold over this syntax is a catamorphism.

def foldSKI[A](
    term: SKI
)(
    s: A,
    k: A,
    i: A,
    app: (A, A) => A
): A =
  term match
    case SKI.S => s
    case SKI.K => k
    case SKI.I => i
    case SKI.App(l, r) =>
      app(foldSKI(l)(s, k, i, app), foldSKI(r)(s, k, i, app))

This is the right abstraction for syntax-directed operations: computing the size of a term, translating it, pretty-printing it, collecting variables, or compiling it. For instance:

def size(term: SKI): Int =
  foldSKI(term)(
    s = 1,
    k = 1,
    i = 1,
    app = (m, n) => 1 + m + n
  )

But execution is not merely such a fold. To execute a term, we must reduce it. For SKI, the rules are:

I x       -> x
K x y     -> x
S x y z   -> x z (y z)

Reduction sends a raw syntax tree to a reduced state. Let us write this abstractly as \(\beta : L \to R\) where L is the language of raw program texts and R is the set of reduced states. The reduced application of two reduced states is then not raw application. It is application followed by reduction: \(r \cdot s := \beta(r s)\).

This is the first place where the magma changes. The syntax has raw application. The reduced state space has transported application. In Scala:

trait ReducedCalculus[R]:
  def app(r: R, s: R): R

Here app(r, s) means reduced application, not raw syntax formation.

Programs as states

Once we have reduced states, we can stop thinking of a program as an inert syntax tree. A reduced program is now a state of a machine. From such a state, there are two things we can ask.

  1. What is currently visible?
  2. WHat happens if I feed it an argument?

This suggests a decomposition \(r \mapsto (hd(r), s \mapsto r \cdot s)\) where \(hd(r)\) is the observable head label of the reduced state \(r\), and \(s \mapsto r \cdot s\) is the transition function that sends an input argument to the next reduced state.

This is a coalgebra. More precisely, if \(B\) is the set of observable head labels, the relevant functor is the Moore functor \(F(X) = B × X^R\). An \(F\)-coalgebra is a map \(X \to B \times X^R\). In our case \(X = R\), so the coalgebra is a function \(\gamma : R \to B \times R^R\) where \(\gamma(r) = (hd(r), s \mapsto r \cdot s)\).

In Scala this becomes:

trait HeadCoalgebra[R, B]:
  def head(r: R): B
  def step(r: R, arg: R): R

  def observe(r: R): (B, R => R) =
    (head(r), (arg: R) => step(r, arg))

This is the coalgebraic half of applicative computation. A catamorphism consumes finite syntax. This coalgebra unfolds future behaviour.

A very small SKI evaluator

To keep the example concrete, let us implement a tiny SKI reducer. This is not meant to be efficient or complete as an evaluator for all purposes. It is only meant to show how reduced application gives the coalgebraic transition.

enum Term:
  case S
  case K
  case I
  case App(left: Term, right: Term)

import Term.*

def app(f: Term, x: Term): Term =
  App(f, x)

We can reduce by repeatedly contracting the outermost SKI redexes.

def reduceOnce(t: Term): Option[Term] =
  t match
    case App(I, x) =>
      Some(x)

    case App(App(K, x), y) =>
      Some(x)

    case App(App(App(S, x), y), z) =>
      Some(App(App(x, z), App(y, z)))

    case App(l, r) =>
      reduceOnce(l).map(App(_, r))
        .orElse(reduceOnce(r).map(App(l, _)))

    case _ =>
      None

Then normalize with a fuel bound to avoid nontermination.

def normalize(t: Term, fuel: Int = 1000): Term =
  if fuel <= 0 then t
  else
    reduceOnce(t) match
      case Some(u) => normalize(u, fuel - 1)
      case None    => t

The reduced product is now:

def reducedApp(r: Term, s: Term): Term =
  normalize(App(r, s))

This gives the transition part of the coalgebra.

Head labels

We next choose what we want to observe. For SKI, a simple head observation records whether the reduced term begins with S, K, I, an application-headed form, or something unknown.

enum Head:
  case HS
  case HK
  case HI
  case HApp

A simple head function is:

def head(t: Term): Head =
  normalize(t) match
    case S        => Head.HS
    case K        => Head.HK
    case I        => Head.HI
    case App(_,_) => Head.HApp

Now the head/currying coalgebra is:

object SKICoalgebra extends HeadCoalgebra[Term, Head]:
  def head(r: Term): Head =
    _root_.head(r)

  def step(r: Term, arg: Term): Term =
    reducedApp(r, arg)

Given a reduced state \(r\), the coalgebra exposes

val (h, next) = SKICoalgebra.observe(r)

where h is the current head and next is the function sending an argument to the next state. This is the computational meaning of currying. A multi-input program is observed one argument at a time. For example, the combinator K has the behaviour

K x y -> x.

Coalgebraically, this is not one ternary operation. It is a sequence of unary transitions:

K      --x-->   K x
K x    --y-->   x

The coalgebra unfolds this process.

Complete behaviour

The one-step coalgebra only tells us the current head and the next-state function. To know the complete observable behaviour of a state, we unfold it along every finite list of arguments.

Mathematically, the terminal coalgebra for \(F(X) = B \times X^R\) is \(B^{R^*}\) where \(R^*\) is the set of finite lists of reduced arguments. A complete behaviour is a function \(R^*\to B\). It sends each finite argument list to the head observed after applying that list.

For a reduced state r, the terminal behaviour is

Theta(r)(s1, ..., sn) = head(((r · s1) · s2) ... sn).

In Scala, for a fixed finite list, we can compute this finite observation.

def runArgs(r: Term, args: List[Term]): Term =
  args.foldLeft(normalize(r)) { case (state, arg) =>
    reducedApp(state, normalize(arg))
  }

def observeAfter(r: Term, args: List[Term]): Head =
  head(runArgs(r, args))

For example:

val x = I
val y = K

val kxy = observeAfter(K, List(x, y))
// K I K -> I, so the observed head is HI.

The full terminal behaviour Theta(r) is not a finite object in general. It is the function that knows the answer for every finite list of arguments.

Computationally, we approximate it by testing finite argument lists.

def behaviourSample(
    r: Term,
    tests: List[List[Term]]
): Map[List[Term], Head] =
  tests.map(args => args -> observeAfter(r, args)).toMap

This gives a finite approximation to the terminal head profile.

Behavioural equivalence

Two reduced states are behaviourally equivalent if they have the same complete head profile. That is, \(r \equiv_{\text{hd}} t\) when \(\Theta(r) = \Theta(t)\).

Unwound, this means:

\[ hd(r \cdot s_1 \cdot \cdots \cdot s_n) = hd(t \cdot s1 \cdot \cdots \cdot s_n) \]

for every finite list of reduced arguments. In Scala we cannot test all lists, but we can test finite samples.

def behaviourallyEqualOn(
    r: Term,
    t: Term,
    tests: List[List[Term]]
): Boolean =
  tests.forall(args => observeAfter(r, args) == observeAfter(t, args))

Let us compare I with a more complicated term that behaves like I. In SKI, the term S K K behaves as identity:

S K K x -> K x (K x) -> x.

Define:

val SKK: Term =
  App(App(S, K), K)

Now test it against I.

val tests =
  List(
    Nil,
    List(I),
    List(K),
    List(S),
    List(App(K, I)),
    List(App(App(S, K), K)),
    List(I, K),
    List(K, I)
  )

val sameOnTests =
  behaviourallyEqualOn(I, SKK, tests)

The term I and the term S K K are not the same syntax. They are not the same tree. But they have the same identity behaviour. A sufficiently compatible head observation should identify them behaviourally. This is the key shift:

syntax equality     compares trees;
reduction equality  compares reduced representatives;
behaviour equality  compares all future observations.

From one calculus to another

The point of this coalgebraic view is not only to compare two programs inside one calculus. It also lets us compare whole calculi. The lambda calculus, SKI, and iota calculus have different syntax. But they can present the same applicative behaviours. A translation between calculi should preserve:

  1. application,
  2. reduction,
  3. head observation,
  4. behavioural indistinguishability.

For example, bracket abstraction translates lambda terms to SKI terms. Conversely, SKI combinators have lambda interpretations:

I := lambda x. x
K := lambda x. lambda y. x
S := lambda x. lambda y. lambda z. x z (y z)

If the translations preserve and reflect terminal head behaviours, then the calculi are equivalent in the behavioural sense. That is stronger and cleaner than saying merely that one syntax can encode another. It says that the two calculi determine the same behaviour space after quotienting by head indistinguishability. In categorical language, one obtains a category of calculi and then inverts the translations that induce bijections on behavioural quotients. This gives a Gabriel–Zisman localization of the category of calculi.

The slogan is:

different syntax, same terminal behaviour.

Recursion as diagonal closure

There is another point where the coalgebraic perspective clarifies the structure. In an applicative calculus, every reduced program a determines a representable endomap \(L_a(x) = a \cdot x\). A fixed point of \(a\) is a program \(p\) such that \(a \cdot p = p\). This is the abstract form of recursion.

The usual Y combinator is one way to produce such fixed points. But the construction can be stated without mentioning the lambda calculus specifically. The reduced magma is diagonally closed if, for every a, there is a program d_a satisfying \(d_a \cdot x = a \cdot (x \cdot x)\) for every \(x\). Then define

\[ p_a = d_a \cdot d_a. \].

We get

\[ p_a = d_a \cdot d_a = a \cdot (d_a \cdot d_a) = a \cdot p_a. \]

So \(p_a\) is a fixed point of \(a\). This is Lawvere’s diagonal argument in the language of reduced applicative magmas. It is the abstract reason that untyped lambda calculus, SKI, and iota support general recursion. Coalgebraically, a fixed point equation \(a \cdot p = p\) turns into an equation of behaviours: \(\Theta(p) = \partial_p \Theta(a)\). This says that the behaviour of \(p\) is obtained by feeding \(p\) back into the behaviour of \(a\). Thus recursion is an algebraic fixed point internally, but it becomes a self-referential behaviour externally.

Where catamorphisms stop and coalgebras begin

The earlier post was about magmas and catamorphisms. The point was syntax is inductive and syntax-directed computation is a fold. That remains true. Parsing, pretty-printing, compilation, translation, and de Bruijn conversion are naturally catamorphic or recursively defined over syntax. But execution is different. A reduced program is not merely a tree to fold. It is a state whose future is probed by arguments. The correct structure is:

\[ \text{state} \rightarrow \text{observation}\times \text{next-state-function}. \]

That is a coalgebra. So the two halves fit together as follows. On the algebraic side we have:

  • build syntax
  • fold syntax
  • translate syntax
  • normalize binding

and on the coalgebraic side we have

  • reduce to states
  • observe heads
  • apply arguments
  • unfold behaviours
  • quotient by indistinguishability

Or more compactly catamorphisms consume syntax but anamorphisms unfold behaviour. That division is the basis for comparing calculi. Lambda calculus, SKI, and iota have different syntax, but their standard translations can identify the same terminal behaviours. The algebraic side explains how programs are built and transformed. The coalgebraic side explains how reduced programs are observed and compared.

A final Scala sketch

Here is the whole idea in a compact interface.

trait ApplicativeCalculus[Raw, Red, Head]:
  def reduce(raw: Raw): Red
  def rawApp(m: Raw, n: Raw): Raw

  def reducedApp(r: Red, s: Red): Red =
    reduce(rawApp(embed(r), embed(s)))

  def embed(r: Red): Raw

  def head(r: Red): Head

  def step(r: Red, s: Red): Red =
    reducedApp(r, s)

  def observe(r: Red): (Head, Red => Red) =
    (head(r), (s: Red) => step(r, s))

  def observeAfter(r: Red, args: List[Red]): Head =
    head(args.foldLeft(r)(step))

The method observe is the head/currying coalgebra \(R \to B \times R^R\). The method observeAfter computes finite pieces of the terminal semantics as \(\Theta(r) : R^* \to B\). The behavioural equivalence relation is equality of these terminal behaviours.

Of course, in an actual implementation one cannot compare all functions \(R^* \to B\). One approximates by finite tests, or proves equivalence mathematically. But the abstraction is still useful: it tells us what the exact semantic object is.
